# Hermes Approval Policy

**Purpose:** Define how approvals work across missions, task packets, commits, and pushes. Approval is never reusable.

---

## Approval Model

```
One Mission
    │
    ├── Task Packet A  (Hermes Direct)
    ├── Task Packet B  (Codex CLI)
    └── Task Packet C  (OpenCode)
            │
            └── One Approval → One Commit → One Push approval
```

Each mission may spawn multiple task packets. Each task packet requires its own execution approval. Each commit requires its own commit approval. Each push requires its own push approval. **Approval cannot be reused.**

---

## Approval IDs

Every approval must be recorded with an Approval ID:

| ID Format | Example | Purpose |
|-----------|---------|---------|
| `MISSION-YYYYMMDD-NNN` | `MISSION-20260710-001` | Identifies the mission |
| `SESSION-YYYYMMDD-NNN` | `SESSION-20260710-001` | Identifies the runtime session |
| `APPROVAL-COMMIT-NNN` | `APPROVAL-COMMIT-001` | Identifies a specific commit approval |
| `APPROVAL-PUSH-NNN` | `APPROVAL-PUSH-001` | Identifies a specific push approval |

## Approval Types

### Execution Approval

Required before an executor begins work.

| Risk Level | Required By | When |
|------------|-------------|------|
| LOW | Por | After Hermes classifies and plans |
| MEDIUM | Por | After Hermes classifies and plans |
| HIGH | Por + ChatGPT | Before any work begins |
| CRITICAL | Por + ChatGPT | Before any work begins |

### Commit Approval

Required before `git commit`.

| Risk Level | Required By | When |
|------------|-------------|------|
| LOW | Por | After Hermes review |
| MEDIUM | Por | After Hermes review |
| HIGH | Por + ChatGPT | After Hermes review |
| CRITICAL | Por + ChatGPT | After Hermes review |

### Push Approval

Required before `git push`. **Always requires explicit Por approval regardless of risk level.**

| Risk Level | Required By | When |
|------------|-------------|------|
| ALL | Por | After commit is accepted |

---

## Approval Rules

| Rule | Description |
|------|-------------|
| **Approval cannot be reused** | Each commit requires its own approval. You cannot batch-approve multiple commits. |
| **Commit ≠ Push** | Approving a commit does not imply push approval. Push always requires a separate explicit command from Por. |
| **Approval is per-session** | An approval given in one Hermes session does not carry over to another session. |
| **Emergency bypass** | If Por directly instructs Codex/OpenCode/DS (Hermes bypass), the bypass itself is considered the execution approval. Commit and push still require explicit Por approval. |
| **Evidence before approval** | Hermes must present full evidence packet before Por gives commit or push approval. |
| **Approval can be revoked** | If new evidence or risks emerge after approval, Por may revoke approval and require re-review. |

## Approval Tokens (Future)

Approval Tokens will be a future enhancement. They will:

- Be generated by Hermes for each approval gate
- Be included in task packets as proof of approval
- Be verified before commit/push
- Expire after the session ends

**Current state:** Not implemented. All approvals are given explicitly by Por in chat.

## Cost Awareness in Approvals

When Por reviews an approval request, Hermes must include:

| Cost Factor | Detail |
|-------------|--------|
| **Preferred cheapest adequate route** | What was the cheapest route that could handle this task? |
| **Codex quota preserved?** | Was Codex quota saved (cheaper route used) or spent (Codex needed for depth)? |
| **Paid executor justified?** | If OpenCode/DeepSeek was used, why was Hermes Direct or Codex read-only insufficient? |
| **Codex quota status** | Evidence-based: remaining limit and reset time, or "unknown — evidence required" |

---

## Summary

1. **One mission, many packets** — each packet needs its own execution approval.
2. **One approval, one commit** — approval cannot be reused.
3. **Commit ≠ Push** — commit approval is separate from push approval.
4. **Push always requires explicit Por** — no autonomous push at any risk level.
5. **Evidence before approval** — Hermes must present full evidence before Por decides.
6. **Cost awareness** — every approval request must include cost/quota justification.
